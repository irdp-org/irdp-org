## 2) บั๊กที่รู้แล้ว ยังไม่แก้ — แก้ก่อนอย่างอื่น

### บั๊ก #1 (จากเฟส 2): หน้ายื่นคำขอ offsite กับหน้า GPS check-in ไม่ได้ต่อกัน
**อาการ:** ทดสอบจริงพบว่า "สามารถยื่นขออนุมัติได้เลย ไม่เห็นมีให้เช็ค GPS อะไรเลย กล้องก็ไม่มีให้ถ่ายรูป อยู่นอกหรือในรัศมีก็ยื่นได้" — แปลว่าฟอร์ม "ยื่นคำขอ" (ส่วน A ของเฟส 2) ทำงานได้ แต่ฟีเจอร์ "เช็คอินจริง" (ส่วน B ของเฟส 2 ที่ควรมี GPS+เซลฟี่+รูปหน้างาน) ดูเหมือนไม่ได้ถูกเรียกใช้หรือเชื่อมเข้ากับ flow หลัก หรือผู้ใช้หาทางเข้าไม่เจอใน UI

**สิ่งที่ควรเป็น (ตามสเปกเดิม):** การ "ยื่นคำขอ" ควรเป็นแค่ขั้นแรก (บอกแผน/ขออนุมัติ) ส่วนการ "เช็คอินจริง" ต้องทำแยกหลังจากนั้น (เมื่อ field_request นั้น status='approved' แล้ว) โดยต้องผ่าน GPS + ถ่ายรูป (เซลฟี่ + รูปหน้างาน ตาม `work_locations.required_photos`) เข้า `attendance_checkins` — ไม่ใช่แค่กรอกฟอร์มแล้วจบเลย

**สิ่งที่ต้องทำ:**
1. ตรวจโค้ดจริงว่าหน้า "เช็คอิน" (`attendance_checkins` insert flow) มีอยู่จริงไหม ถ้ามีแต่หาทางเข้าไม่เจอ → เพิ่มปุ่ม/ลิงก์ที่ชัดเจนในหน้ารายละเอียดคำขอ (เช่น ปุ่ม "เช็คอินตอนนี้" ที่โผล่เมื่อ status='approved' และยังไม่มี checkin วันนี้)
2. ถ้ายังไม่มีเลย (ทำแค่ A ไม่ได้ทำ B จริง) → สร้างใหม่ตามสเปกเดิมในส่วน B ของ `PHASE2_KICKOFF_PROMPT.md` (ดูไฟล์เก่าถ้ามี หรือดู IRDP_System_Architecture.md ส่วน 6.2)
3. ต้องคำนวณระยะทาง (haversine) ฝั่ง **server** เทียบกับ `work_locations` ของคำขอนั้น ไม่ใช่ client คำนวณแล้วส่งค่า distance มาตรงๆ (กันปลอม GPS)
4. ไม่บล็อกการเช็คอินถ้านอกรัศมี (ตามสเปก HR ตรวจด้วยตา) แต่ต้อง flag `within_radius=false` ให้เห็นชัดในหน้าอนุมัติ/รีวิว

## 4) กฎสำคัญที่ต้องรักษาไว้ (ย้ำจาก CLAUDE.md เผื่อลืม)

- **Business logic ของ OT/วันลา/กันชนเวลาจอง อยู่ใน DB (Postgres functions/triggers/RLS) ทั้งหมด** — ห้ามเขียนตรรกะคำนวณซ้ำในโค้ด TypeScript ฝั่งแอป ไม่ว่าจะเฟสไหน
- **service_role key เคยหลุดไป git history มาก่อน 1 ครั้งแล้ว** (rotate + scrub history ไปแล้ว) — เน้นย้ำ: ห้าม commit `.env.local` หรือ secret ใดๆ ขึ้น git อีกเด็ดขาด เช็ค `.gitignore` ทุกครั้งที่สงสัย
- **Vercel แผน Hobby** — cron จำกัดแค่วันละครั้ง; งานที่ต้องการความถี่สูงกว่านั้น (Google Calendar pull-sync, WFH reminder) ใช้ **GitHub Actions ยิงเข้า API route ที่ guard ด้วย `CRON_SECRET`** แทน (ตั้งไว้แล้วในเฟส 2) — ถ้าจะเพิ่ม cron ใหม่ ให้ใช้ pattern เดียวกันนี้ ไม่ต้องเดาทำใหม่
- **migration ใหม่ทุกไฟล์ห้ามแก้ของเดิมย้อนหลัง** สร้างไฟล์ใหม่ตามเลขลำดับถัดไปเสมอ และรันผ่าน Supabase SQL Editor/CLI เท่านั้น (ห้ามรันผ่าน MCP บน production)
- **Design system:** สีแบรนด์ `#283897`(navy)/`#F36523`(orange) จากโลโก้จริง, ฟอนต์ IBM Plex Sans Thai/Sans, โทนสว่าง Notion×iOS, มือถือเป็นหลัก, bottom tab bar — ห้าม hardcode สีใหม่นอกเหนือ CSS variables ที่ตั้งไว้ในเฟส 0
- **บัญชีที่ใช้:** GitHub/Vercel/Supabase/Google Cloud ทั้งหมดใช้ `irdpofficer@gmail.com` (แยกจากบัญชีส่วนตัวที่ใช้กับโปรเจกต์อื่นชื่อ proppsy) — **ระมัดระวังเป็นพิเศษทุกครั้งที่เข้า Supabase/Vercel dashboard ว่าอยู่ใน org/project ของ IRDP จริง ไม่ใช่ proppsy** (เคยพลาดดู project ผิดมาแล้วรอบหนึ่งตอนจะ rotate key)
- **Google OAuth consent screen เป็น External + published (production)** ไม่ใช่ Testing mode แล้ว — ถ้าเจอปัญหา auth ใหม่ ไม่ใช่เพราะ test user list อีกต่อไป ให้สงสัยที่อื่นก่อน (เช่น redirect URI, env vars)
- **พนักงานในระบบจริงตอนนี้มีคนเดียว** (อานนท์ แย้มโชติ, admin, ฝ่ายธุรการ) — ฟีเจอร์ที่ต้องทดสอบกับคนหลายคน (อนุมัติข้ามกัน, แจ้งเตือนหาหัวหน้าฝ่าย, คนขับรถ) ต้องเพิ่มพนักงานทดสอบก่อนจะเห็นผลจริง (ใช้หน้า `/admin/employees` ที่ทำไว้ในเฟส 1.5)
- **รถตู้ส่วนกลางยังไม่ตั้ง driver_id** — ต้อง `UPDATE vehicles SET driver_id = '<employee_id>' WHERE name = 'รถตู้ส่วนกลาง';` เองหลังมีพนักงานคนที่จะเป็นคนขับเข้าระบบแล้ว
